# ExtBlocks: всё, что происходит вокруг основного ответа

Основная модель в Glaze отвечает за текст сцены. Но иногда одной прозы недостаточно: после ответа нужно обновить статус персонажа, создать иллюстрацию, выполнить автоматизацию или показать интерактивную панель.

Для этих задач существует `ExtBlocks` — система дополнительных блоков, работающих вокруг обычной генерации.

Удобная метафора: **основная модель — актёр на сцене, а ExtBlocks — закулисная команда**. Пока актёр произносит реплику, команда может подготовить справку, нарисовать декорацию, запустить механизм или развернуть панель управления.

ExtBlocks не являются частью Studio preset. У них свои пресеты, настройки, API-профили, разрешения и результаты.

## Что умеют ExtBlocks

В актуальной реализации есть четыре типа блоков:

| Тип | Что делает |
|---|---|
| **Infoblock** | Создаёт дополнительный текст или HTML. |
| **Image** | Подготавливает инструкцию и запускает генерацию изображения. |
| **JS Runner** | Создаёт или выполняет JavaScript в изолированной среде. |
| **Interactive Panel** | Показывает интерактивную HTML-панель под сообщением. |

Результаты обычно отображаются отдельно от основной прозы. По умолчанию они не возвращаются в промпт будущих ответов.

## Пресет ExtBlocks

`ExtensionPreset` — это набор блоков, которые должны работать вместе. В нём хранятся:

- название;
- порядок блоков;
- параметры каждого блока;
- профили LLM API;
- разрешения для JavaScript.

Одновременно активен не более одного пресета. По умолчанию ExtBlocks выключены и активный пресет не выбран.

Выбор пресета в настройках автоматически включает расширение. Выбор пункта **«Не выбран»** убирает активный пресет, но сам общий переключатель может остаться включённым.

## Общие настройки блока

Каждый блок определяет:

- тип и название;
- включён ли он;
- момент запуска;
- позицию в цепочке;
- нужно ли ждать предыдущий блок;
- сколько сообщений истории передать;
- сколько старых результатов того же блока добавить;
- какой API использовать;
- показывать ли результат только в панели или также добавить его в будущий промпт;
- разрешён ли автоматический запуск.

По умолчанию новый блок:

- включён;
- запускается после ответа ассистента;
- видит 10 последних сообщений до точки запуска;
- не зависит от предыдущего блока;
- не инжектируется в основной промпт;
- может запускаться автоматически.

## Infoblock

**Инфоблок** создаёт дополнительный текст или HTML на основе истории чата и пользовательской инструкции.

Например, он может показать:

- состояние персонажей;
- краткое содержание сцены;
- список предметов;
- показатели отношений;
- справку о текущей локации;
- любой другой сопровождающий материал.

### Как он работает

1. Glaze собирает историю вокруг сообщения, к которому привязан запуск.
2. При необходимости добавляет старые результаты этого же блока.
3. Передаёт модели инструкцию и шаблон.
4. Получает текст или HTML.
5. Пытается извлечь содержимое из ожидаемых тегов.
6. Сохраняет результат и показывает его под сообщением.

Стандартный шаблон использует XML-подобную оболочку с названием блока. Если модель не вернула ожидаемые теги, непустой ответ всё равно может быть сохранён целиком. Поэтому шаблон помогает направить и разобрать результат, но не является строгой схемой валидации.

## Image

Блок **Image** разделяет работу между текстовой моделью и генератором изображений.

```text
История и инструкция
        |
        v
LLM создаёт описание изображения
        |
        v
Image Gen создаёт пиксели
        |
        v
PNG сохраняется и показывается в блоке
```

Сначала LLM-агент создаёт специальную инструкцию, обычно помеченную как `[IMG:GEN]`. Затем Image Gen provider получает описание, стиль, соотношение сторон и размер изображения.

Используется первая подходящая инструкция из ответа. Готовый PNG сохраняется локально в каталоге `generated` и заменяет служебный маркер ссылкой на результат.

Если глобальная генерация изображений выключена, текстовый этап всё равно может завершиться: HTML или инструкция сохранятся, но пиксельное изображение создано не будет.

Для готового image-блока доступна отдельная команда **«Картинка»**. Она повторяет только генерацию пикселей и не просит LLM заново придумывать описание.

При включённом image context новый запрос может получить до трёх последних готовых изображений ExtBlocks в качестве контекста.

## JS Runner

`JS Runner` выполняет JavaScript в изолированной среде. Он поддерживает два сценария.

### JavaScript от LLM

Модель получает задачу и возвращает код. Glaze извлекает JavaScript, запускает его и сохраняет возвращённое значение как содержимое блока.

Если в ответе модели не найден код, блок завершается ошибкой.

### Готовый скрипт

Если LLM-инструкция пуста, но задано поле `script`, готовый JavaScript выполняется напрямую без обращения к модели.

Скрипт получает контекст с:

- сообщениями;
- идентификаторами сессии и персонажа;
- именем и описанием персонажа;
- его характером и сценарием;
- результатом предыдущего блока, если он был передан по цепочке.

Через API `window.glaze` скрипт также может работать с разрешёнными переменными, запускать генерацию, временно добавлять данные в промпт, воспроизводить звук, выполнять команды и показывать уведомления.

## Interactive Panel

**Интерактивная панель** — это HTML-интерфейс, открываемый под ответом ассистента.

HTML может быть:

- написан заранее;
- сгенерирован LLM при запуске блока.

Панель работает в отдельном sandboxed `iframe`. Она может сообщить Glaze о готовности, изменить свою высоту, закрыться или отправить действие через разрешённый мост.

Это позволяет создавать:

- карточки состояния;
- кнопки выбора;
- мини-интерфейсы;
- панели управления сценой;
- небольшие интерактивные виджеты.

HTML панели сохраняется как обычный результат ExtBlocks. Однако после полного пересоздания WebView сохранённая интерактивная панель не обязана автоматически восстановиться как активный iframe: её содержимое останется в базе и обычной панели результатов.

## Когда запускаются блоки

### After User

Блок стартует сразу после сохранения сообщения пользователя, параллельно генерации основного ответа.

Основной ответ не ждёт его завершения. Поэтому результат `After User` не гарантированно успеет повлиять на текущую генерацию.

Результат сохраняется у пользовательского сообщения, но отдельная панель под такими сообщениями намеренно не показывается. Обычный механизм prompt injection ищет результаты у сообщений ассистента, поэтому `After User` напрямую через него не инжектируется.

### After Assistant

Это основной режим. Блок запускается после получения ответа ассистента и привязывается к его текущему свайпу.

При включённой Studio запуск происходит после выбора итоговой версии текста, чтобы инфоблок или изображение относились именно к тому ответу, который видит пользователь.

### Periodic

В модели конфигурации предусмотрен периодический запуск JavaScript по таймеру.

Однако в актуальном приложении scheduler не подключён к постоянному жизненному циклу интерфейса, поэтому нельзя считать `Periodic` работающей пользовательской функцией. В редакторе также нет полноценной настройки интервала, а в dormant-реализации остаются несогласованности параметров.

Пока этот trigger следует воспринимать как заготовку для будущего подключения.

### Только вручную

Блок с флагом **«Только вручную»** исключается из автоматической цепочки. Его можно запускать индивидуально из панели.

Команда общего запуска использует автоматический процессор `After Assistant`, поэтому такие блоки в общий Run All также не входят.

## Контекст блока

История всегда считается относительно **anchor message** — сообщения, на котором висит блок, а не относительно конца чата.

Настройка количества сообщений работает так:

- `0` — не передавать историю;
- `-1` — передать всю историю до anchor включительно;
- положительное число — передать указанное количество последних сообщений до anchor.

Блок может дополнительно получить:

- несколько прошлых результатов блока с тем же именем;
- `previousOutput` от непосредственного предыдущего блока цепочки;
- данные персонажа и активной персоны.

## Макросы

В инструкциях и шаблонах доступны макросы:

- `{{char}}` — имя персонажа;
- `{{user}}` — имя активной персоны;
- `{{description}}` — описание персонажа;
- `{{personality}}` — его характер;
- `{{scenario}}` — сценарий карточки.

Если данных нет, макрос заменяется пустой строкой.

В пути prompt injection надёжно доступна активная персона, но объект персонажа сейчас не передаётся полностью. Поэтому часть character-макросов при повторной инъекции может раскрыться не так, как при первоначальном запуске блока.

## Порядок и зависимости

Перед запуском блоки сортируются по полю `order`.

Независимые блоки могут выполняться параллельно. Это сокращает общее время, если, например, инфоблок и изображение не используют результаты друг друга.

Флаг **«Ждать завершения предыдущего блока»** создаёт простую цепочку:

```text
Блок A
   |
   | previousOutput
   v
Блок B
```

Зависимый блок ждёт только непосредственного соседа перед ним. Это не полноценный граф зависимостей: нельзя выбрать произвольный block ID или построить сложную топологию.

Если предыдущий блок завершился ошибкой, его сохранённый HTML с сообщением об ошибке всё равно может попасть в `previousOutput` следующего блока.

## ExtBlocks и Studio

### Studio выключена

После основного ответа ExtBlocks запускаются в фоне. Чат не обязан удерживать кнопку Stop до окончания всех дополнительных блоков.

Результаты привязываются к верхнему свайпу ответа без отдельного Studio sub-swipe.

### Studio включена

Сначала Studio определяет итоговый текст:

- очищенную версию;
- исходную версию, если Cleaner ничего не изменил;
- fallback или частичный результат при соответствующем сценарии.

Только после этого ExtBlocks привязываются к выбранной паре `swipeId` и `agentSwipeId`. Благодаря этому изображение или панель не остаются прикреплёнными к черновой версии, когда пользователь видит очищенную.

В некоторых ранних ветках завершения Cleaner ExtBlocks могут не запуститься. Поэтому включённая Studio не является абсолютной гарантией запуска каждого блока.

## Свайпы и результаты

Каждый обычный результат ExtBlocks хранит:

- сессию;
- сообщение;
- верхний свайп;
- внутренний Studio-свайп;
- идентификатор, имя и тип блока;
- содержимое;
- порядок и статус;
- время создания.

При переключении ответа панель ищет результаты именно для текущей пары свайпов. Для старых данных поддерживается fallback к legacy-варианту без внутреннего Studio-свайпа.

## Prompt injection

По умолчанию у блока установлено:

```text
inject = false
```

Это означает, что результат:

- сохраняется;
- показывается пользователю;
- не влияет на будущую прозу.

Такой режим особенно безопасен при использовании Studio Canon: пользовательские панели не смешиваются с канонической памятью.

Если включить **«Инжектировать в промпт»**, результат добавляется к одному или нескольким последним сообщениям ассистента. Каждый блок прикрепляется к тому сообщению, для которого был создан. Между ответом и блоком можно добавить пользовательский префикс.

Интерфейс отдельно предупреждает, что такой режим не рекомендуется вместе со Studio Canon.

### Runtime injection из JavaScript

`glaze.injectPrompt` — другой механизм. Он не связан с полем `inject` у блока.

JavaScript может временно создать prompt-block с:

- идентификатором;
- содержимым;
- ролью `system`, `user` или `assistant`;
- глубиной размещения.

Такие блоки хранятся только в памяти приложения и исчезают после перезапуска. Размер и идентификаторы ограничены, а само действие требует разрешения пресета.

## JavaScript и безопасность

JavaScript обычно выполняется в невидимом WebView, внутри отдельного iframe с `sandbox="allow-scripts"`. Доступ к same-origin родительского окна не выдаётся.

Если headless engine недоступен, Glaze может использовать видимый chat WebView как fallback.

На выполнение действуют тайм-ауты:

- около 30 секунд для headless engine;
- около 55–60 секунд для визуального fallback.

### Разрешения пресета

По умолчанию потенциально чувствительные возможности запрещены. Пресет отдельно разрешает:

- чтение и изменение переменных чата, персонажа, сообщения или глобальных данных;
- дополнительную генерацию текста;
- запуск основной генерации;
- временную инъекцию в промпт;
- выполнение команд;
- воспроизведение аудио;
- показ уведомлений.

Если проверка разрешения отсутствует, применяется правило **default deny**.

### Честная модель безопасности

Sandbox защищает от прямого доступа к родительскому WebView и API-ключам. Но разрешённые методы `glaze.*` специально создают мост наружу.

Кроме того, обычное HTML-содержимое некоторых блоков вставляется в интерфейс через `innerHTML` без строгого универсального sanitizer.

Поэтому ExtBlocks нельзя считать безопасной средой для произвольного недоверенного HTML или JavaScript. Безопасность зависит от sandbox, выданных разрешений, проверок моста и доверия к автору пресета.

## Ошибки и остановка

Блок может находиться в одном из состояний:

- ожидает;
- выполняется;
- готов;
- завершён с ошибкой;
- остановлен.

Ошибка сохраняется как отдельное оформленное содержимое блока, поэтому пользователь видит её прямо в панели.

Автоматического цикла повторных попыток на уровне общей оркестрации нет. Повтор запускает пользователь:

- **«Перегенерировать»** — заново выполнить один блок;
- **«Картинка»** — повторить только Image Gen;
- **«Запустить блоки»** — повторить набор `After Assistant`.

Кнопка Stop выглядит относящейся к одному блоку, но сейчас отменяет все активные ExtBlocks-задачи через общий набор cancel tokens.

После перезапуска приложения оставшиеся записи со статусом `running` переводятся в `stopped`.

## Хранение и синхронизация

Пресеты и результаты ExtBlocks хранятся в локальной базе данных. Настройка включения и идентификатор активного пресета сохраняются отдельно.

Presets и InfoBlocks входят в архитектуру резервного копирования и облачной синхронизации.

Сгенерированные изображения сохраняются отдельными PNG-файлами, а в результате блока остаётся ссылка на локальный путь.

## Важные ограничения

- `Periodic` предусмотрен моделью, но сейчас не активирован приложением как рабочий scheduler.
- `After User` работает параллельно ответу и не гарантирует влияние на текущую генерацию.
- Результаты `After User` не показываются под пользовательским сообщением и не инжектируются обычным assistant-механизмом.
- Run All выполняет цепочку `After Assistant`, а не все существующие trigger-типы.
- Блоки **«Только вручную»** не входят в Run All.
- Переключатель стриминга Interactive Panel пока не подключён к LLM callback.
- Начальная высота Interactive Panel не сохраняется в конфигурации блока.
- Поле `imageGenEnabled` существует, но runtime не использует его как отдельный выключатель.
- HTML не проходит строгую универсальную sanitization.
- Ошибка предыдущего блока может стать входом следующего зависимого блока.
- Интерактивный iframe не гарантированно восстанавливается после перезагрузки WebView.
- Остановка одного блока сейчас может остановить остальные активные блоки.
- Prompt injection не учитывает внутренний Studio-свайп столь же строго, как панель результатов, поэтому при сложной истории sub-swipes возможны неоднозначности.

## Глоссарий

| Термин | Определение |
|---|---|
| **ExtBlocks / Ext Blocks** | Система дополнительных задач и панелей вокруг основной генерации чата. |
| **ExtensionPreset** | Набор блоков, API-профилей и JavaScript-разрешений. |
| **BlockConfig** | Настройки одного блока: тип, trigger, порядок, контекст и поведение. |
| **Infoblock** | LLM-сгенерированный дополнительный текст или HTML. |
| **Image agent** | Текстовая модель, создающая инструкцию для Image Gen. |
| **Image Gen** | Провайдер, превращающий текстовое описание в изображение. |
| **JS Runner** | Генерация или выполнение JavaScript в sandbox. |
| **Interactive Panel** | HTML-интерфейс в отдельном sandboxed iframe. |
| **Trigger** | Событие, запускающее блок. |
| **After User** | Запуск после сохранения сообщения пользователя, параллельно ответу. |
| **After Assistant** | Запуск после финализации ответа ассистента. |
| **Periodic** | Предусмотренный таймерный trigger для JS; в текущем приложении не активирован. |
| **Manual only** | Режим, исключающий блок из автоматической цепочки. |
| **Anchor message** | Сообщение, относительно которого собирается история блока. |
| **previousOutput** | Содержимое непосредственного предыдущего блока цепочки. |
| **previousBlocksCount** | Количество старых результатов блока с тем же именем. |
| **Dependency** | Ожидание предыдущего блока и передача его результата следующему. |
| **Prompt injection** | Добавление сохранённого результата ExtBlocks к истории ассистента. |
| **Runtime prompt injection** | Временный prompt-block, созданный через `glaze.injectPrompt`. |
| **Capability** | Явное разрешение пресета на действие JavaScript. |
| **Default deny** | Запрет действия, если разрешение не было явно выдано. |
| **Headless engine** | Невидимый WebView для выполнения JavaScript. |
| **Visual fallback** | Выполнение JavaScript через chat WebView при недоступности headless engine. |
| **Sandbox** | Ограниченная среда исполнения HTML или JavaScript. |
| **swipeId** | Идентификатор выбранного верхнего варианта ответа. |
| **agentSwipeId** | Идентификатор внутреннего варианта Studio, например очищенного текста. |
| **Cleaner** | Редакторская фаза Studio, после которой выбирается итоговый текст. |
| **InfoBlock row** | Сохранённый в базе результат одного запуска блока. |
| **Rerun** | Ручной повтор выполнения блока. |
