# MemoryBook: как Glaze помнит длинные истории

Даже у большой языковой модели есть ограниченное окно контекста. Пока история короткая, в него можно положить почти весь чат. Но со временем старые сцены вытесняются новыми, важные детали теряются, а каждый следующий ответ становится всё дороже.

`MemoryBook` решает эту проблему с помощью отдельной книги воспоминаний. Он превращает выбранные фрагменты переписки в короткие редактируемые записи, а перед генерацией следующего ответа находит среди них наиболее подходящие.

Если представить чат книгой, то:

- сообщения — полная стенограмма истории;
- черновик памяти — закладка на нескольких страницах;
- активная запись — утверждённый пересказ эпизода;
- поиск памяти — библиотекарь, выбирающий нужные заметки;
- инъекция памяти — передача этих заметок автору следующей сцены.

MemoryBook не пытается постоянно показывать модели всю историю. Вместо этого он отвечает на более практичный вопрос: **какие события из прошлого полезно вспомнить именно сейчас?**

## Что хранит MemoryBook

У каждой чат-сессии своя книга памяти. В ней находятся:

- подтверждённые записи о прошлых эпизодах;
- черновики, ожидающие генерации или проверки;
- ключевые слова для поиска;
- ссылки на исходные сообщения;
- настройки отбора и упаковки памяти.

Запись может содержать встречу героев, обещание, конфликт, открытие, путешествие или любой другой важный фрагмент истории.

MemoryBook — это **эпизодическая память**. Он лучше всего отвечает на вопрос «что раньше произошло?», но не является единственным источником истины о текущем состоянии мира.

## Путь одного воспоминания

### 1. Выбор фрагмента истории

Система делит старую часть переписки на диапазоны сообщений. Самые свежие реплики временно остаются нетронутыми, чтобы незавершённая сцена не превратилась в преждевременный пересказ.

По умолчанию один черновик охватывает 15 сообщений, а последние 4 сообщения остаются в этой «зоне отставания». Неполный диапазон ждёт продолжения истории.

Уже обработанные сообщения повторно не превращаются в новые черновики.

### 2. Создание черновика

Черновик хранит диапазон и ссылки на исходные сообщения, но сначала не содержит готового воспоминания.

Он может появиться двумя способами:

- автоматически после очередного ответа;
- вручную после команды **«Сканировать чат»**.

На этом этапе LLM ещё ничего не пересказывает. Система только отмечает подходящий фрагмент истории.

### 3. Генерация пересказа

По команде пользователя отдельная модель получает выбранные сообщения и создаёт:

- короткое содержание эпизода;
- набор ключей, по которым его можно будет найти.

Черновик переходит в состояние ожидания проверки. Его можно прочитать и отредактировать до добавления в постоянную память.

Генерация черновика не запускается одновременно с генерацией основного ответа в той же сессии. Это предотвращает конфликт двух параллельных LLM-задач.

### 4. Подтверждение

Кнопка **«Принять»** превращает черновик в активную запись MemoryBook. Вместе с текстом сохраняется его происхождение: диапазон и идентификаторы исходных сообщений.

Если включён векторный поиск, новая запись также отправляется на индексацию.

До подтверждения черновик не участвует в поиске памяти. Пользователь остаётся редактором книги и решает, что считать достойным долговременного хранения.

```text
Сообщения
    |
    v
Черновик диапазона
    |
    v
LLM-пересказ
    |
    v
Проверка пользователем
    |
    v
Активная запись MemoryBook
```

## Ручная и автоматическая работа

После каждого ответа Glaze в фоне может:

- подготовить новые пустые черновики MemoryBook;
- создать векторный индекс исходных сообщений для Raw Recall.

При этом создание диапазона и генерация его пересказа — разные операции. Текущий автоматический этап создаёт черновики, но не заполняет их текстом через LLM. Переключатель **«Автогенерация черновиков»** присутствует в настройках, однако в актуальном пайплайне автоматическое заполнение к нему не подключено.

В окне **«Книги памяти»** пользователь может:

- сканировать чат;
- добавлять запись вручную;
- генерировать и редактировать черновики;
- принимать или удалять черновики;
- редактировать и удалять активные записи;
- переиндексировать записи;
- удалить векторные индексы;
- искать дубликаты;
- отфильтровать память по выбранным свайпам.

## Как выбираются воспоминания

Перед генерацией система собирает короткий поисковый запрос из текущего текста, недавних реплик и, при соответствующей настройке, последнего ответа ассистента.

Затем активные записи получают оценку релевантности. Для этого могут использоваться сразу несколько сигналов.

### Ключевые слова

У каждой записи есть ключи: имена, места, предметы, события или темы. Если они встречаются в текущем контексте, запись получает дополнительный вес.

Доступны разные способы сопоставления: простое вхождение, поиск целых слов Glaze или сочетание обоих вариантов.

### Векторный поиск

Векторный поиск сравнивает не отдельные слова, а общий смысл текстов. Благодаря этому запись о «разговоре в больнице» может найтись по запросу о «посещении палаты», даже если точные формулировки не совпадают.

Для него необходимы:

- настроенный embedding endpoint;
- индекс записей;
- совместимая модель эмбеддингов.

Сам отбор выполняется локально после одного запроса за embedding. Дополнительная LLM для решения «какую память выбрать» не запускается.

### Каталог

В сбалансированном режиме Glaze строит локальный каталог записей. В него входят название, ключи, сюжетная линия, важность, диапазон и краткое описание.

Каталог помогает находить записи не только по тексту, но и по связанным сущностям, местам и темам.

### Дополнительные сигналы

На итоговую оценку также могут влиять:

- свежесть записи;
- её важность;
- совпадение персонажей и сущностей;
- эмоциональная близость;
- разнообразие выбранного набора.

После подсчёта система выбирает лучшие записи, пока не достигнет ограничения по количеству и доступному бюджету контекста.

## Защита от повторов

Если исходные сообщения записи уже напрямую видны модели в текущем окне чата, MemoryBook может не вставлять их пересказ повторно. Эта функция называется **Source-window exclusion**, или **«Пропускать уже видимые источники»**.

Так модель не получает одну и ту же сцену одновременно в полном и сокращённом виде, а место в контексте остаётся для более старых воспоминаний.

Если хотя бы одно исходное сообщение записи уже видно, запись может быть исключена целиком. Это простое и предсказуемое правило, но при частичном пересечении оно иногда скрывает больше памяти, чем необходимо.

## Бюджет памяти

Найти полезные записи недостаточно: их ещё нужно уместить в контекст модели.

Эффективный бюджет MemoryBook ограничивается:

- долей от общего окна контекста;
- абсолютным лимитом, если он задан.

По умолчанию память может занять до 35% контекста. В интерфейсе доступны готовые размеры и пользовательский лимит.

### Полные записи

В режиме **«Полные»** система вставляет только целые воспоминания. Если запись не помещается, она пропускается.

Этот режим сохраняет связность текста, но может использовать бюджет менее эффективно.

### Гибрид

В режиме **«Гибрид»** сначала вставляются целые записи, а для оставшегося места выбираются релевантные выдержки из более крупных воспоминаний.

Это стандартный компромисс между связностью и экономией контекста.

### Чанки

В режиме **«Чанки»** система ранжирует отдельные фрагменты записей. Он позволяет извлечь точную деталь из большого воспоминания, не передавая модели весь пересказ.

Выбранные фрагменты всё равно выводятся в хронологическом порядке, чтобы прошлое не превращалось в случайный набор цитат.

## Как память попадает в промпт

Выбранные воспоминания собираются в отдельный блок `Memory Book`. В нём явно указано, что это ранее принятые свидетельства из текущей сессии.

Есть два основных способа размещения:

- **Hard Block** — Glaze автоматически добавляет отдельное system-сообщение перед историей;
- **`{{memory}}`** — автор пресета сам выбирает место для памяти с помощью макроса.

Если выбран режим `{{memory}}`, но в пресете нет соответствующего макроса или блока памяти, собранные воспоминания не попадут в запрос. Интерфейс показывает диагностическое предупреждение, но не переносит их автоматически.

## Raw Recall: память без пересказа

Помимо MemoryBook, в Glaze есть `Raw Recall`. Это похожая по назначению, но отдельная система.

После генерации чат индексируется небольшими группами исходных сообщений. Перед следующим ответом Raw Recall ищет семантически похожие фрагменты и возвращает их почти дословно.

| MemoryBook | Raw Recall |
|---|---|
| Краткий редактируемый пересказ | Почти дословный фрагмент чата |
| Требует активной записи | Работает по индексу сообщений |
| Использует ключи, каталог и векторы | Использует семантическое сходство |
| Курируется пользователем | Работает автоматически |
| Экономит больше контекста | Лучше сохраняет исходные детали |

Raw Recall служит страховкой от потерь при пересказе. Он также не повторяет сообщения, уже видимые в текущем окне.

MemoryBook можно отключить, не отключая Raw Recall: эти механизмы собираются и запускаются отдельно.

## MemoryBook и канон Studio

В Glaze память разделена на несколько слоёв.

### MemoryBook: что произошло

MemoryBook хранит эпизодические пересказы прошлого. Запись может быть релевантной одной сцене и не понадобиться в другой.

### Studio Ledger: что истинно сейчас

`Studio Ledger` поддерживает компактное текущее состояние сцены, мира, отношений, NPC и сюжетных арок.

Ledger не пересказывает всю историю. Он отвечает на вопрос: **что считается актуальным каноном в данный момент?**

### Character Knowledge Facts: кто что знает

Отдельные атомарные факты описывают знания, убеждения и развитие персонажей. Они учитывают, кому принадлежит знание и из какого ответа оно появилось.

```text
MemoryBook             -> что произошло раньше
Raw Recall             -> как это было сказано в чате
Character Facts        -> кто что знает и как изменился
Studio Ledger          -> что истинно в мире сейчас
```

При конфликте текущий канон Ledger и знания персонажей важнее эпизодического пересказа. MemoryBook, в свою очередь, важнее противоречащих ему исходных сведений карточки персонажа.

## Связь со Studio

### Когда Studio включена

MemoryBook подбирается для контекстных окон Studio так же, как для обычной генерации. После финального текста Ledger может обновить текущий канон и знания персонажей.

Создание черновиков MemoryBook и индексация Raw Recall остаются отдельными фоновыми задачами.

### Когда Studio выключена

Обычная модель продолжает получать MemoryBook и Raw Recall. Новые черновики также продолжают создаваться.

Ledger в этом режиме не обновляется, но уже принятый канон и активные факты персонажей не удаляются и могут по-прежнему попасть в промпт.

## Происхождение и свайпы

MemoryBook хранит связь записи с исходными сообщениями и диапазоном истории. Это позволяет:

- понимать происхождение воспоминания;
- фильтровать записи по выбранным свайпам;
- удалять память вместе с удалённым источником;
- не дублировать уже обработанные диапазоны.

У Ledger защита свайпов строже: новое состояние сначала считается предварительным и становится активным только после продолжения истории с выбранного ответа.

У MemoryBook подтверждение выполняет сам пользователь через кнопку **«Принять»**.

## Важные ограничения

MemoryBook помогает сохранять непрерывность, но не гарантирует идеальную память.

- Пересказ выполняется с потерями и может пропустить важную деталь.
- LLM может ошибиться при составлении черновика, поэтому предусмотрено подтверждение человеком.
- Автоматическое создание черновика не означает автоматическую генерацию его текста.
- Векторный поиск требует отдельного embedding endpoint и актуальных индексов.
- После смены embedding-модели старые векторы могут стать несовместимыми.
- Ошибка поиска обычно просто оставляет запрос без найденной памяти и не останавливает чат.
- Режим `{{memory}}` требует правильно размещённого макроса в пресете.
- Слишком большой бюджет памяти может вытеснить значительную часть обычной истории.
- Некоторые служебные поля, включая важность, сюжетную линию и временную слепоту, пока не редактируются в основном интерфейсе записи.
- Флаг блокировки записи MemoryBook не является полной защитой от всех операций изменения и объединения.
- Dedup запускается вручную и требует векторы и LLM.
- Memory Graph и часть эвристического анализа сейчас не наполняются обычным post-turn пайплайном.

Главное правило: **MemoryBook — это полезная редакторская память, а не безошибочная база данных фактов.**

## Глоссарий

| Термин | Определение |
|---|---|
| **MemoryBook / Книга памяти** | Сессионная книга подтверждённых эпизодических пересказов. |
| **MemoryDraft / черновик** | Диапазон сообщений, ещё не ставший активной записью. |
| **MemoryEntry / запись памяти** | Принятое воспоминание, доступное системе поиска. |
| **Active** | Состояние записи, разрешающее retrieval и инъекцию в промпт. |
| **Retrieval** | Поиск подходящих воспоминаний перед генерацией. |
| **Keyword retrieval** | Поиск памяти по заданным ключевым словам. |
| **Vector retrieval** | Семантический поиск по числовому представлению смысла текста. |
| **Embedding** | Числовой вектор, описывающий смысл текста для поиска похожих фрагментов. |
| **Catalog** | Локальный индекс названий, ключей, сущностей и тем MemoryBook. |
| **Score** | Итоговая оценка полезности записи для текущего запроса. |
| **Packing** | Способ уложить выбранную память в доступный контекст. |
| **Chunk / чанк** | Отдельный фрагмент большой записи или группа исходных сообщений. |
| **Excerpt / выдержка** | Выбранная часть более крупного воспоминания. |
| **Token budget** | Максимальный объём контекста, отведённый памяти. |
| **Source window** | Часть истории, которую модель уже видит напрямую. |
| **Source-window exclusion** | Исключение памяти, источник которой уже находится в видимой истории. |
| **Hard Block** | Автоматическое system-сообщение с выбранной памятью. |
| **`{{memory}}`** | Макрос для ручного размещения MemoryBook в пресете. |
| **Raw Recall** | Семантическое возвращение почти дословных старых сообщений. |
| **Studio Ledger** | Компактное актуальное состояние канона сессии. |
| **Character Knowledge Fact** | Отдельный факт о знании, отношении или развитии персонажа. |
| **Provenance** | Связь записи с сообщениями и свайпами, из которых она получена. |
| **Swipe** | Выбранный вариант ответа модели. |
| **Importance** | Дополнительный вес записи при ранжировании. |
| **Arc** | Связанная сюжетная линия; в MemoryBook это метаданные записи. |
| **Temporally blind** | Запись, полезность которой не уменьшается из-за возраста. |
| **Dedup** | Ручной поиск и объединение похожих записей. |
| **Continuity guard** | Инструкция не выдумывать прошлое, если надёжная память не найдена. |
